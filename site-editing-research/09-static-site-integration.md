# Static Site Integration (Hugo / Jekyll / 11ty / Astro / Gatsby / Hexo)

Static sites are nearly all on Git. The integration is therefore the **GitHub App** (see `05-github-integration.md`) plus framework-aware file detection. This document focuses on the framework-aware bits: where each generator looks for content, frontmatter, layouts, and config.

## Detection Heuristics

| Generator | Tell-tale files |
|-----------|-----------------|
| Hugo | `config.toml` / `hugo.toml` / `hugo.yaml`; `content/`; `themes/` |
| Jekyll | `_config.yml`; `_posts/`, `_layouts/`; `Gemfile` with `jekyll` |
| 11ty | `.eleventy.js` or `eleventy.config.js`; `_includes/` |
| Astro | `astro.config.mjs`; `src/content/`; `package.json` has `astro` |
| Gatsby | `gatsby-config.js`; `src/pages/`, `src/templates/` |
| Hexo | `_config.yml` + `package.json` has `hexo` |
| MkDocs | `mkdocs.yml`; `docs/` |
| Docusaurus | `docusaurus.config.js`; `docs/`, `blog/` |

Surven runs detection by reading the repo's root file list via the GitHub Contents API.

## Common Pattern: Frontmatter Edits

All these generators use YAML/TOML frontmatter at the top of `.md`/`.mdx` files. Surven's pipeline:

1. Read file via Contents API.
2. Parse with `gray-matter` (preserves whitespace).
3. Mutate frontmatter object.
4. Re-serialize, preserving original key order and quoting style.
5. Write back.

Editable frontmatter keys per generator:

### Hugo
```yaml
title: ...
description: ...
date: ...
lastmod: ...
draft: false
images: ["/og/post.png"]   # used by OpenGraph templates
canonicalURL: ...
aliases: [...]   # for redirects
```

### Jekyll
```yaml
title: ...
description: ...
date: ...
permalink: ...
image: ...
author: ...
seo:
  type: BlogPosting
```

### 11ty
```yaml
title: ...
description: ...
permalink: ...
date: ...
tags: [...]
eleventyExcludeFromCollections: false
```

### Astro Content Collections
Astro's content collections enforce a Zod schema in `src/content/config.ts`. Surven reads the schema, validates proposed frontmatter, then writes. Schema-aware editing is a Surven advantage — invalid writes fail at preview time.

### Gatsby
Frontmatter consumed by GraphQL queries in templates. Edits are safe.

## Layouts and Partials

Site-wide head injection (schema, OG defaults, llms.txt link, Surven analytics) goes in:

| Generator | File | Pattern |
|-----------|------|---------|
| Hugo | `layouts/partials/surven-head.html` + included from `head.html` | One partial owned by Surven |
| Jekyll | `_includes/surven-head.html` + included from `_includes/head.html` | Same |
| 11ty | `_includes/surven-head.njk` | Same |
| Astro | `src/components/SurvenHead.astro` imported in layout | Same |
| Gatsby | `gatsby-ssr.js` with `setHeadComponents` | API hook |
| Docusaurus | swizzle `Layout/index.js` | Avoid; prefer `customFields` + plugin |

Surven's "isolation rule" applies: own one file. Never directly edit shared layouts. The first-time setup PR adds an include line to the shared layout, then never touches it again.

## Static Files (Public Root)

All generators serve files from a `static/` or `public/` directory at the build root.

| Generator | Static dir |
|-----------|------------|
| Hugo | `static/` |
| Jekyll | site root (with `permalink`) or `assets/` |
| 11ty | `public/` (configurable) |
| Astro | `public/` |
| Gatsby | `static/` |
| Next.js (static export) | `public/` |

Surven writes:
- `static/llms.txt`
- `static/llms-full.txt`
- `static/robots.txt` (or merges with existing)
- `static/og/<slug>.png` (generated OG images)
- `static/sitemap.xml` (if not generated by a plugin)

## Sitemaps

Most generators have a sitemap plugin:
- Hugo: built-in (`config.toml` `sitemap`).
- Jekyll: `jekyll-sitemap` gem.
- 11ty: `eleventy-plugin-sitemap` or hand-written.
- Astro: `@astrojs/sitemap` integration.
- Gatsby: `gatsby-plugin-sitemap`.

Surven detects which is in use; if none, ships one in the PR.

## Robots.txt

Same pattern. Surven uses fenced edits (`# surven-managed-block-start` / `end`) so the existing robots.txt is preserved.

## Build Step Awareness

Surven detects deploy target from CI config:
- `.github/workflows/*.yml` mentioning `vercel`, `netlify`, `cloudflare/pages-action` → host known.
- `vercel.json`, `netlify.toml`, `wrangler.toml` → host known.
- No CI but `gh-pages` branch → GitHub Pages.

Knowing the host matters because:
- Vercel/Netlify deploy preview URLs are used for diff preview screenshots.
- Cache headers may need adjusting for `llms.txt` (text/markdown content type).
- Some hosts (Cloudflare Pages) honor `_headers` and `_redirects` files; Surven uses these instead of editing config files when present.

## Schema Injection on Static Sites

Per-page schema goes in frontmatter as a `schema:` block consumed by a layout helper Surven adds. Site-wide Organization schema lives in the `surven-head` partial.

## Idempotency

Trivial: file content is the canonical state. Surven hashes the desired content, compares to current. Skip if equal.

## Rollback

PR revert. That's the whole story for static sites — Git handles it.

## Special Cases

### Docusaurus
Frontmatter-heavy. SEO defaults in `docusaurus.config.js`. Per-page in MDX frontmatter. Surven supports both.

### MkDocs
YAML config at `mkdocs.yml`; navigation defined there. Markdown files have minimal frontmatter (only `title`).

### Notion-as-CMS, Headless WP for Static, Sanity-driven Astro
Detect via package.json deps (`@notionhq/client`, `@wordpress/*`, `@sanity/client`). When detected, edits go to the **CMS**, not the static repo. Cross-reference `10-headless-cms-integration.md`.

## Failure Modes

| Failure | Recovery |
|---------|----------|
| Build fails after Surven PR (broken frontmatter) | Surven runs a local build pre-PR via the framework's CLI in a sandbox container |
| Plugin missing (e.g., site has no sitemap plugin) | PR adds the plugin to package.json/Gemfile |
| Custom theme doesn't include head partial | Surven asks for confirmation before editing theme files |
| Massive site (10k+ pages) blows past GitHub Contents API limits | Use Git Data API (tree + commit) for atomic batch |

## Build Performance

For sites with thousands of pages, Surven batches frontmatter edits into a single commit (many file changes, one PR). This avoids one-PR-per-page noise.
