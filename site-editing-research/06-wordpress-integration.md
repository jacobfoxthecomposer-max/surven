# WordPress Integration

WordPress powers ~43% of the web. For Surven, it is the highest-volume CMS by a wide margin and the second integration to ship after GitHub. This document covers self-hosted WordPress (the hard case); WordPress.com is a strict subset.

## Connection Method: REST API + Application Passwords

WordPress 5.6+ ships **Application Passwords** built-in. They are user-scoped tokens that grant REST API access without exposing the user's real password. They are revocable individually and have a name + creation date for audit.

### Connection flow

1. Client clicks "Connect WordPress" in Surven.
2. Surven asks for site URL.
3. Surven verifies REST is reachable (`GET /wp-json` returns 200 with discovery JSON).
4. Surven uses the Application Passwords authorization endpoint:
   `https://<site>/wp-admin/authorize-application.php?app_name=Surven&success_url=<callback>&reject_url=<callback>`.
5. Client logs in (if needed) and clicks Approve.
6. WP redirects to `<callback>?user_login=...&password=...&site_url=...`.
7. Surven encrypts the password with a per-tenant KEK and stores in Supabase.

This is far better UX than asking clients to manually generate Application Passwords from `wp-admin/profile.php`.

### Capability requirements

The connecting WP user must have:
- `edit_posts`, `edit_pages` — for content edits.
- `edit_others_posts` — to edit posts by other authors.
- `manage_options` — to update site settings (only required for site-wide robots.txt and llms.txt).
- `upload_files` — to add OG images.

Recommend a dedicated **"Surven Editor" role** Surven creates via its companion plugin — this is cleaner than asking the client to use an Administrator account.

## What's Editable via REST

| Surface | Endpoint | Notes |
|---------|----------|-------|
| Post content | `POST /wp/v2/posts/{id}` | Block editor compatible. |
| Page content | `POST /wp/v2/pages/{id}` | |
| Post meta (any) | Same endpoint, `meta` field | Requires `show_in_rest` registration. |
| Featured image | `featured_media` field | Use media ID. |
| Categories/tags | `categories`, `tags` arrays | |
| Media alt text | `POST /wp/v2/media/{id}` `alt_text` | |
| Custom post types | `POST /wp/v2/<cpt>/{id}` | Requires `show_in_rest: true` on CPT. |

## Yoast SEO

**Yoast's REST API is read-only as shipped.** Listing endpoint exposes Yoast data on GET, but PUT/POST do not write. To write:

- **Option A (recommended):** Surven ships a tiny companion plugin that registers `_yoast_wpseo_title`, `_yoast_wpseo_metadesc`, `_yoast_wpseo_canonical`, `_yoast_wpseo_opengraph-title`, `_yoast_wpseo_opengraph-description`, `_yoast_wpseo_opengraph-image`, and `_yoast_wpseo_twitter-*` meta keys with `show_in_rest: true` and an auth_callback that requires `edit_post` capability.
- **Option B:** Use a third-party (e.g., `chazuk/wp-api-yoast-meta`). Avoid — third-party drift is a support burden.

After registration, write via:
```http
POST /wp-json/wp/v2/posts/123
{ "meta": { "_yoast_wpseo_title": "...", "_yoast_wpseo_metadesc": "..." } }
```

## RankMath

RankMath stores SEO data in `rank_math_title`, `rank_math_description`, `rank_math_focus_keyword`, `rank_math_canonical_url`, `rank_math_robots`. Same `show_in_rest` registration requirement. The community `rank-math-api-manager` plugin exposes them at `/wp-json/rankmath/v1/updateMeta`. Surven's companion plugin handles both Yoast and RankMath uniformly.

## ACF (Advanced Custom Fields)

ACF Pro 5.11+ supports REST when `show_in_rest: true` is set on the field group. Field values appear in the `acf` key of REST responses and are writable:
```http
POST /wp-json/wp/v2/posts/123
{ "acf": { "hero_headline": "...", "hero_cta": "..." } }
```
Use case: Surven can rewrite hero copy on landing pages built with ACF without touching the page builder UI.

## Robots.txt and Sitemap

WordPress generates a virtual robots.txt by default. To customize without a plugin, hook `robots_txt` in PHP — Surven's companion plugin does this and reads its config from a wp_option Surven manages via REST.

Sitemap is auto-generated by WP core (since 5.5) at `/wp-sitemap.xml`. Yoast/RankMath override with their own sitemap.

## llms.txt

WP core does not serve arbitrary text files at the root. Two paths:
1. Companion plugin registers a virtual `/llms.txt` route that reads from a wp_option.
2. Yoast added native llms.txt support in 2025 — Surven detects Yoast version and defers to it when present.

## Block Editor Considerations

Gutenberg stores content as HTML with block delimiters (`<!-- wp:paragraph -->`). When Surven edits content:
- **Preserve all block delimiters** — stripping them breaks the visual editor.
- **Use the `freeform` block** when injecting raw HTML that doesn't match a known block.
- **Schema injection** goes via header/footer scripts (companion plugin), NOT via post content.

## Revisions and Rollback

Every `POST /wp/v2/posts/{id}` creates an autosave or revision. WP retains a configurable number (default unlimited).

- Surven stores the revision ID returned from the response (`_links.version-history.0.href`).
- One-click rollback: `POST /wp-json/wp/v2/posts/{id}/revisions/{rev_id}/restore` — note this is **not** a core endpoint. Surven's companion plugin provides it.

## Page Builders (Elementor, Divi, Beaver Builder, WPBakery)

These store layout in serialized post meta, not the `content` field. Editing post `content` does not affect what's rendered.

- **Elementor:** data in `_elementor_data` post meta as JSON. Editable via REST if registered, but changing it requires deep knowledge of Elementor's schema.
- **Divi:** uses shortcodes in `content`. Edits to `content` do work, but shortcode parsing is fragile.
- **Surven posture for v1:** detect page-builder pages and skip content edits on them; only do head/meta/schema fixes via the companion plugin's wp_head injection. Page-builder content edits are a future add-on.

## WooCommerce

Products are a custom post type (`product`). Writable via `/wp/v2/product/{id}` once `show_in_rest: true` (WooCommerce already does this). Product short_description and description editable. Product attributes editable via WC REST API (`/wc/v3/products/{id}`), which uses Consumer Key/Secret rather than Application Passwords — Surven supports both.

## Multisite

`installation_id`-equivalent here is per-site. Surven treats each site in a network as a separate connection, indexed by `blog_id`.

## Failure Modes

| Failure | Recovery |
|---------|----------|
| 401 Unauthorized | Application Password revoked or user deactivated. Trigger reconnection flow. |
| 403 (capability) | User lacks `edit_post` for that post. Surface to client with explanation. |
| 503 (host blocking REST) | Many shared hosts (esp. WP Engine, Kinsta) firewall REST writes from unknown IPs. Surven publishes its IP range so clients can allowlist. |
| Slow site (>30s) | Stream the request; if timeout, queue and retry off-peak. |
| Plugin conflict (security plugin blocks REST) | Detect via test write; instruct client to allowlist Surven's user agent. |
| Post locked by another user | Force unlock if Surven is the editor; respect lock if it's a human. |

## Companion Plugin

Surven ships a minimal WP plugin (~300 LOC) that handles:
1. Registers Yoast/RankMath/ACF meta with `show_in_rest`.
2. Provides `/wp-json/surven/v1/restore-revision/{post_id}/{rev_id}`.
3. Hooks `wp_head` to inject Surven-managed schema/OG blocks.
4. Hooks `robots_txt` to merge Surven's allow rules.
5. Provides virtual `/llms.txt` and `/llms-full.txt` routes.
6. Logs every Surven write to a custom table for audit.
7. Self-uninstall: deactivation removes all Surven hooks and data.

The plugin should be open-source on GitHub for trust and submitted to the WP Plugin Directory for one-click install.

## Pricing / Limits

- WP Engine and other managed hosts impose REST rate limits (~5/sec common).
- Surven serializes per-site writes and respects 429 with backoff.
