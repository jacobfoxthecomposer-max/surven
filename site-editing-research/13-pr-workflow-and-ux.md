# PR Workflow and Approval UX

This document covers branch strategy, commit messages, PR templates, status checks, auto-merge rules, and the in-Surven approval UX. It is GitHub-specific in detail but the principles apply to draft-publish flows on Shopify themes, Webflow drafts, and CMS drafts.

## Branch Strategy

Pattern: `surven/<fix-id>/<short-slug>`.

Examples:
- `surven/llms-txt/initial`
- `surven/og-tags/about-page`
- `surven/answer-capsule/pricing-page`
- `surven/internal-links/orphan-batch-2026-05`

Why this pattern:
- `surven/` namespace makes Surven branches obvious in branch lists.
- `<fix-id>` is the canonical fix identifier (e.g., `llms-txt` = fix #17).
- `<short-slug>` disambiguates multiple PRs of the same fix type.

Branches are deleted on merge (auto-delete via repo setting Surven recommends but doesn't force).

## Commit Messages

Conventional Commits format with `[surven]` suffix to make Surven commits filterable in `git log`:

```
feat(seo): add llms.txt for AI crawler discovery [surven]
fix(seo): correct canonical URL on /pricing [surven]
chore(seo): add loading=lazy to below-fold images [surven]
```

Atomic: one logical change per commit. Multi-file changes that belong together (e.g., adding llms.txt requires updating robots.txt) ship as a single commit.

## PR Title

Plain English. No `[surven]` suffix in title (it's already labeled). One line, ≤72 chars when possible.

Examples:
- `Add llms.txt for AI crawler discovery`
- `Add OpenGraph tags to homepage`
- `Rewrite homepage answer-capsule for "GEO agency" query`

## PR Body Template

```markdown
## What this PR does

<one-paragraph plain-English description>

## Why this matters for AI visibility

<one-paragraph GEO rationale; cites which engine/query benefits>

## Files changed

- `public/llms.txt` (new) — root-level file consumed by ChatGPT, Claude, Perplexity crawlers.
- `public/robots.txt` (modified) — added llms.txt sitemap reference.

## Preview

- Vercel preview: <auto-injected URL>
- Before/after screenshots: <Surven-hosted URLs>

## Re-measurement

Surven will re-scan this site within 48 hours of merge and report whether AI mentions, citations, or rank positions changed. View results at <surven-dashboard-url>.

## Rollback

Click "Revert" on this PR to undo. Surven will detect the revert and re-add the fix to your queue.

---
Generated by [Surven](https://surven.com) — fix ID `llms-txt`. Run ID: `<run_id>`. [Open in Surven](<deep-link>)
```

PR body is generated server-side and posted via GitHub API. `<deep-link>` opens the run in the Surven dashboard for context.

## Labels

Surven applies these labels (creating them if they don't exist):
- `surven` — every Surven PR.
- `surven:tier-1` / `surven:tier-2` / `surven:tier-3`.
- `surven:auto-merge` — when applicable.
- `surven:re-measure` — after merge, before re-scan.

## Status Checks

Surven respects the repo's existing CI:
- Wait for all required status checks to pass before enabling auto-merge.
- If any check fails, post a comment with diagnostic info, do NOT auto-merge.
- Surven adds its own check `surven/preview` that turns green when the deploy preview URL is reachable and the change renders correctly (headless-Chrome smoke test).

## Auto-Merge Rules

Per-tenant configuration:
- **Off (default):** every PR requires a human merge in GitHub or in Surven UI.
- **Tier-1 only:** auto-merge enabled for Tier-1 fixes after CI passes.
- **Tier-1 + first-pass Tier-2:** auto-merge enabled after the client has reviewed the first 3 of any Tier-2 fix type ("trust ladder").
- **Off for specific paths:** never auto-merge changes to paths matching `*.config.*`, `next.config.js`, etc.

GitHub's auto-merge feature requires:
- Branch protection enabled on the base branch.
- Auto-merge enabled in repo settings.
- Surven enables via `enablePullRequestAutoMerge` GraphQL mutation.

If auto-merge can't be enabled (free org, branch protection off), Surven falls back to "merge button" UX in the Surven dashboard.

## Approval UX (Surven Dashboard)

The Surven dashboard surfaces every open PR with:

```
┌──────────────────────────────────────────────────────────┐
│ Add OpenGraph tags to /pricing                  [Tier 1] │
│                                                          │
│ Why: ChatGPT cannot generate a thumbnail when citing     │
│ this page, reducing visibility in AI Overviews.          │
│                                                          │
│ ┌── Before ─────────┐  ┌── After ──────────┐             │
│ │ <screenshot>      │  │ <screenshot>      │             │
│ └───────────────────┘  └───────────────────┘             │
│                                                          │
│ Code diff:                                               │
│ + <meta property="og:title" content="Surven Pricing">    │
│ + <meta property="og:description" content="...">         │
│ + <meta property="og:image" content="/og/pricing.png">   │
│                                                          │
│ [ Reject ] [ Edit before merge ] [ Approve & merge ]     │
└──────────────────────────────────────────────────────────┘
```

Clicking "Approve & merge" merges via GitHub API (no need to leave Surven). Clicking "Edit" opens a Monaco editor with the diff; saved edits become a force-push to the Surven branch.

## Visual Diff Generation

For Tier-2/3 fixes that change rendered content, Surven generates before/after screenshots:

1. Wait for Vercel/Netlify preview deploy success.
2. Headless Chrome navigates to production URL, screenshots full page.
3. Headless Chrome navigates to preview URL, screenshots full page.
4. Side-by-side render in Surven UI; pixel-diff highlights changes in red.

For Shopify themes (no preview URL): use Shopify's preview theme URL (`?preview_theme_id=`). For Webflow (no preview URL for content edits): show source diff only.

## Conflict Resolution

When Surven opens a PR and the base branch advances:
1. Auto-rebase Surven's branch on `main`.
2. If clean rebase: continue.
3. If conflict and the conflicting hunk is in a Surven-managed marker block: prefer Surven's version (we own it).
4. If conflict and the conflicting hunk is outside Surven's markers: pause the PR, open a GitHub issue tagging the integration installer, label `surven:conflict`.

Manual resolution required for case 4. Surven does NOT silently overwrite human changes.

## Manual-Edit Detection

Surven listens to `push` webhooks. If a push lands on the default branch and modifies a file Surven manages (matched via path patterns + marker presence), Surven:
1. Re-fetches the file.
2. Compares to last-known Surven state.
3. If the human edit is inside the Surven marker block: log a "drift" event, surface in dashboard, do not regenerate.
4. If the human edit is outside the marker: ignore (it's their change).

This prevents Surven from undoing human work — a class of bug that destroys trust.

## Bulk PR Strategy

Some fixes naturally span many files (alt text, frontmatter dates, internal links). Surven groups by fix type:
- One PR per fix-type per "batch."
- Batch size capped at 50 file changes per PR (reviewability).
- For sites with >50 changes of the same type, Surven opens N PRs and chains them with PR-body cross-references.

## Per-Platform Equivalents

| Platform | "PR" equivalent |
|----------|-----------------|
| GitHub | Pull Request |
| GitLab | Merge Request |
| Bitbucket | Pull Request |
| Shopify | Draft theme + preview URL |
| Webflow | Draft CMS items + manual publish |
| WordPress | Draft post + preview |
| Contentful | Entry update + manual publish |
| Sanity | Draft document in Studio |
| Ghost | Draft post |

The Surven approval UX is uniform across platforms — only the underlying merge mechanism differs.
